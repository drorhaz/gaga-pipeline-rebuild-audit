## 1. Executive Summary

Nine external repositories were inspected against targeted extraction criteria. All nine were found locally at `ref_repo_git/` (downloaded as ZIP archives — no `.git` directories, so no remote URL or commit hash is available).

Three repositories yield **directly actionable lessons** for the Gaga pipeline:

1. **kineticstoolkit** — confirms and codifies the NaN-safe filter pattern: detect → warn → temporary fill → filter → restore NaN. This exact distinction (temporary interpolation vs. true repair) is missing from our pipeline logs.
2. **pose2sim** — confirms the value of per-filter config subsections with explicit defaults and a single `type` selector. Also provides a confirmed example of the `zero == missing` hazard we must NOT adopt.
3. **opensim_pyprocessing** — provides a clean typed-config + step-toggle architecture that directly inspires stronger stage contracts in our Papermill-based pipeline.

Five repositories yield **relevant_with_adaptation** or **inspiration_only** lessons:

4. **optitrack-main** — confirms that anti-aliasing should be applied before downsampling in resampling; confirms OptiTrack metadata fields to extract in post-collection QC.
5. **optitrack_motive** — confirms frame-number continuity check as a post-collection QC target (for offline use via parquet frame index, not NatNet).
6. **pyomeca** — confirms clean separation of interpolation from filtering; NaN handling NOT in their filter (hazard for us: our filtering should not rely on external libraries that silently ignore NaN).
7. **scikit-kinematics** — **CRITICAL CONVENTION RISK confirmed**: uses scalar-FIRST `[w, x, y, z]` ordering; our pipeline is SciPy `xyzw` scalar-last. Must not copy any quaternion code.
8. **vaila** — confirms SOS-based Butterworth implementation and adaptive padding as best practices.

One repository is limited in utility: **ezc3d** is primarily an IO library; useful only for future C3D boundary discipline.

**Three key audit questions generated** for Phase 4 deep-dive audits:
- Q-EXT1: Does our pipeline log the NaN count before and after filtering per stage? Does it distinguish temporary interpolation from PCHIP/SLERP true repair?
- Q-EXT2: Does our config distinguish stable defaults, sensitivity options, and experimental options?
- Q-EXT3: Does our post-collection QC script check frame number continuity and metadata fields (frame rate, rotation type, length units)?

### Repository: kineticstoolkit/kineticstoolkit

#### Local inspection

- **Local path:** `ref_repo_git/kineticstoolkit-master`
- **Remote URL:** not available (ZIP download)
- **Files inspected:** `kineticstoolkit/filters.py`, `kineticstoolkit/timeseries.py` (lines 1–150)

#### Relevant logic found

**File:** `kineticstoolkit/filters.py`, lines 40–53 — `_interpolate()` helper

```text
_interpolate(ts, key):
  nan_index = ts.isnan(key)          # 1. detect missing samples
  if not np.all(~nan_index):
    ts = ts.fill_missing_samples(0)  # 2. temporary fill (zeros) — NOT true gap fill
    warnings.warn("NaNs found ...")  # 3. warn user
  return (ts, nan_index)             # returns both filled series AND original NaN map
```

This helper is used in:
- `savgol()` lines 123–138: calls `_interpolate()`, filters, then `filtered_data[nan_index] = np.nan`
- `butter()` lines 259–268: calls `_interpolate()`, filters, then `subts.data[data][missing] = np.nan`
- `smooth()`: delegates to `savgol()`, inherits the same behavior
- **NOT used** by `deriv()` or `median()` — those filters have no NaN handling

`fill_missing_samples(0)` fills with 0, not with interpolated values. This is a minimal fill for filter stability only. The NaN positions are tracked separately and restored after filtering.

`timeseries.py` lines 1–150: `TimeSeries` is a custom data structure not relevant for adoption (our pipeline uses pandas/parquet). The `TimeSeriesInfoDict` and `info["Time"]["Unit"]` pattern shows metadata-aware design.

`_validate_input()` checks: (1) time not empty, (2) sample rate is constant, (3) time unit is 's' (warns if not).

#### Candidate lessons for Gaga pipeline

1. **NaN-safe filter pattern**: detect → warn → temporary fill → filter → restore NaN. Our Stage 3 Butterworth filter does not currently distinguish between PCHIP-repaired frames and originally-missing frames. Adding a `nan_restoration_count` log field would confirm this invariant is being maintained.
2. **Minimum viable NaN handling in filtering**: filling with 0 (not PCHIP) for filter stability is a valid and simpler approach than our current PCHIP repair for short gaps. The key insight is that these are DIFFERENT operations with DIFFERENT provenance.
3. **`_validate_input()` pattern**: pre-filter validation of sample rate consistency and time unit is a useful stage guard. Our NB04 resampling should guarantee constant sample rate; adding an assertion at the start of NB04 filtering is worth adding.

#### Applicability labels

- NaN-safe filter pattern: `directly_relevant`
- TimeSeries data structure: `not_applicable` (do not migrate from pandas)
- Pre-filter sample-rate validation: `relevant_with_adaptation`

#### What not to adopt

- Do NOT migrate to `TimeSeries` data structure.
- Do NOT use `fill_missing_samples(0)` as a substitute for our PCHIP gap fill. These are different operations.
- Do NOT treat NaN-restored frames as "repaired" data — KTK explicitly does NOT claim they are repaired.

#### Concrete audit questions created

- **Q-EXT1a**: Does our pipeline log how many frames were NaN at filter input, and how many NaN frames were in the data after filtering (verifying NaN positions were preserved)?
- **Q-EXT1b**: In Stage 3 (filtering), do we correctly distinguish: (a) frames that were PCHIP-repaired in Stage 1, (b) frames that were Hampel-replaced in Stage 2, and (c) frames that are genuinely missing and should remain NaN through filtering?

### Repository: perfanalytics/pose2sim

#### Local inspection

- **Local path:** `ref_repo_git/pose2sim-main`
- **Remote URL:** not available (ZIP download)
- **Files inspected:** `Pose2Sim/filtering.py` (full), `Pose2Sim/Demo_Batch/Config.toml` (lines 1–100, 215–293)

#### Relevant logic found

**File:** `Pose2Sim/filtering.py` — filter type organization

Nine filter types in a single module: `butterworth`, `acc_minimizing`, `one_euro`, `gcv_spline`, `kalman`, `butterworth_on_speed`, `gaussian`, `loess`, `median`. All exposed via a single dispatcher `filter1d()` with a `filter_type` string parameter (line 755–785).

Each filter function takes `(config_dict, frame_rate, col)` — configuration is always passed as a dictionary, never global.

**`zero == missing` pattern confirmed in Butterworth (lines 538–543):**
```text
mask = np.isnan(col_filtered) | col_filtered.eq(0)
```
This treats zeros as missing. Same pattern in `kalman_filter_1d`, `gcv_spline_filter_1d`, `acc_minimizing_filter_1d`, `butterworth_on_speed_filter_1d`. However, `loess_filter_1d` (line 626) uses `mask = np.isnan(col_filtered)` only — zeros are NOT treated as missing.

This inconsistency within Pose2Sim itself confirms the hazard. In our pipeline, zero IS a valid value (root-relative positions, velocities at rest).

**NaN handling per filter**: each filter splits the signal into sequences of non-NaN (or non-zero for some) values and filters each sequence independently via index slicing. This avoids filter instability at boundaries.

**`recap_filter3d()` function (lines 788–851)**: logs all filter parameters to a human-readable summary. This includes the chosen filter type and all its parameters. Saved to `User/logs.txt`.

**Config.toml pattern (lines 218–261):**
```toml
[filtering]
type = 'butterworth'   # single selector

   [filtering.butterworth]
   cut_off_frequency = 6
   order = 4

   [filtering.kalman]
   trust_ratio = 500
   smooth = true
   ...
```
Each filter type has its own named subsection. Inactive filter sections remain in config but are simply not used. The `type` selector controls which one runs.

**`save_filt_plots = true`**: automatic plot saving enabled by default — filtered vs. unfiltered comparison plots per keypoint.

#### Candidate lessons for Gaga pipeline

1. **Per-filter config subsections with named keys**: our current `config_v1.yaml` has a flat filtering section. Organizing it into `[filtering.butterworth]`, `[filtering.hampel]`, `[filtering.pchip]` style subsections would improve clarity, especially for sensitivity/experimental options.
2. **Single active-type selector + inactive sections**: keeping inactive filter parameters in config (but unused) is a good documentation practice — future sensitivity studies don't require config restructuring.
3. **`recap_filter3d()` style summary log**: our current filtering stage writes JSON summaries. Adding a human-readable log of all filter parameters (type, cutoff, order, adaptive iterations) would improve reproducibility.
4. **Sequence-splitting pattern**: filtering each non-missing sequence independently (instead of the whole array) is already done in our Butterworth stage; confirm it's also done in PCHIP and Hampel stages.

#### Applicability labels

- Per-filter config subsections: `directly_relevant` (candidate for config refactor)
- `zero == missing` hazard: `not_applicable` — confirmed we must NOT adopt this
- recap/summary logging: `relevant_with_adaptation`
- Sequence-splitting pattern: `relevant_with_adaptation`

#### What not to adopt

- Do NOT adopt `zero == missing` logic.
- Do NOT add the full Pose2Sim filter suite (Kalman, LOESS, GCV spline, acc_minimizing, One Euro) unless a specific need is identified. Each new filter option needs justification, tests, and a default-off policy.
- Do NOT use `butterworth_on_speed` — derivative-first filter is inappropriate for quaternion-based processing.
- Do NOT import markerless, OpenSim, or multi-camera assumptions.

#### Concrete audit questions created

- **Q-EXT2a**: Does our `config_v1.yaml` clearly separate: (1) primary parameters (always active), (2) sensitivity options (explicit choice, documented), (3) experimental options (default-off)?
- **Q-EXT2b**: Do we log all filter parameters (type, cutoff, order, method, correction iterations) to a structured field in the filtering summary JSON and to a human-readable line in the notebook output?

### Repository: FIL-OPMEG/optitrack

#### Relevant logic found

**`readRigidBody.m` (lines 56–64):** Reads from OptiTrack CSV header:
```text
configOptions = {
```

These are parsed from the file header and stored in a `cfg` struct. This is exactly the provenance metadata we should extract and log per session.

**`resampleOptiTrack.m` (lines 41–46):** Anti-aliasing before downsampling:
```text
if mean(diff(tfinal)) > mean(diff(tinit))  % if downsampling:
    new_sampling_f = 1/mean(diff(tfinal))
    cutoff_f = new_sampling_f/2            % Nyquist of target rate
    [B,A] = butter(3, 2*cutoff_f/sampling_f)  % anti-aliasing filter
end
```
Anti-aliasing is applied only when downsampling (not upsampling). Our Step 3 resampling should confirm this check.

**`getRigidBodyVelocity.m` (lines 42–55):** Angular velocity via Euler-angle gradient on the Euler angles (NOT quaternion log method):
```text
wx = gradient(deg2rad(X_Rotation), dt)./dt
wy = gradient(deg2rad(Y_Rotation), dt)./dt
wz = gradient(deg2rad(Z_Rotation), dt)./dt
```
This is an Euler-angle finite-difference approach — subject to gimbal lock and Euler sequence dependency. Our pipeline uses the superior quaternion-log method. This confirms our approach is more correct.

**`filterOptitrackData.m`:** Applies FieldTrip Butterworth (low/high/bandpass/bandstop) to rigid body data and markers. No NaN handling — assumes clean data. Applies to all rigid bodies and markers uniformly with the same cutoffs.

**`make_pos_hist.m` (not read, but present):** Creates position histograms — fast visual QC for translation range sanity.

#### Candidate lessons for Gaga pipeline

1. **Provenance metadata from file header**: extract `Capture Frame Rate`, `Export Frame Rate`, `Total Frames in Take`, `Total Exported Frames`, `Rotation Type`, `Length Units`, `Coordinate Space` from C3D/CSV header and log them in the per-run config snapshot.
2. **Anti-aliasing gate check**: verify that Step 3 resampling applies anti-aliasing before downsampling (if going from capture rate to lower processing rate).
3. **Angular velocity method comparison**: the gradient-on-Euler approach confirms our quaternion-log method is more rigorous. Add a note to the kinematics docs.
4. **Visual QC checklist**: translation plots, rotation plots, velocity sanity — confirm our post-collection QC notebook covers these.

#### Applicability labels

- Provenance metadata extraction: `directly_relevant`
- Anti-aliasing gate: `directly_relevant`
- Angular velocity comparison: `relevant_with_adaptation` (confirms our method is better, adds documentation value)
- Full MATLAB port: `not_applicable`

#### What not to adopt

- Do NOT port MATLAB code directly.
- Do NOT use Euler-angle finite-difference angular velocity — our quaternion-log method is superior.
- Do NOT copy synchronization assumptions unless they match our acquisition setup.

#### Concrete audit questions created

### Repository: pyomeca/pyomeca

#### Local inspection

- **Local path:** `ref_repo_git/pyomeca-master`
- **Remote URL:** not available (ZIP download)
- **Files inspected:** `pyomeca/processing/filter.py` (full), `pyomeca/processing/interp.py` (full), `pyomeca/processing/angles.py` (lines 1–60)

#### Relevant logic found

**`filter.py` (full, 62 lines):** Minimal `_base_filter()` function using `scipy.signal.butter` + `filtfilt`. Applies `xr.apply_ufunc(filtfilt, b, a, array)`. **No NaN handling whatsoever.** Pyomeca's filter silently passes NaN through scipy which may produce unexpected results or all-NaN output.

This is a **hazard warning** for our pipeline: any external library we use for filtering must be verified to handle NaN correctly. Pyomeca does not.

**`interp.py` (full, 27 lines):**
- `interpolate_missing_data()`: `array.interpolate_na(dim="time", method="linear")` — linear interpolation, stored in `interp.py` (separate from `filter.py`)
- `time_normalize()`: resamples to fixed N frames — clearly different operation

The separation of `interpolate_missing_data` (in `interp.py`) from filter logic (in `filter.py`) is an explicit design choice: interpolation and filtering are separate concerns, in separate modules.

**`angles.py`:** Standard rotation matrix decomposition into Euler sequences (xyz, xzy, yzx, zxy, zyx, zyz, etc.) via explicit formula. Clean API — `angles_from_rototrans(caller, rt, angle_sequence)`. Uses xarray DataArray dimensions for labeled axes.

#### Candidate lessons for Gaga pipeline

1. **Separation of interpolation from filtering as separate modules/stage outputs**: our pipeline should clearly separate (a) PCHIP gap fill output, (b) Hampel artifact replacement output, (c) filtered output. Currently all three are within the same notebook (NB04). A cleaner provenance trail would log which frames were gap-filled, which were artifact-replaced, and which were filtered from clean data.
2. **NaN-in-filter hazard**: Pyomeca's filter has no NaN handling — confirms we must never use external filter wrappers without verifying NaN safety.

#### Applicability labels

- IO/interpolation/filter separation pattern: `relevant_with_adaptation`
- xarray DataArray: `not_applicable` (do not migrate from pandas/parquet)
- NaN hazard confirmation: `directly_relevant`

#### What not to adopt

- Do NOT migrate to xarray DataArray or Pyomeca data structures.
- Do NOT use Pyomeca's `filter.py` directly — it has no NaN handling.
- Do NOT generalize the pipeline into a broad biomechanics API.

#### Concrete audit questions created

### Repository: thomas-haslwanter/scikit-kinematics

#### Local inspection

- **Local path:** `ref_repo_git/scikit-kinematics-master`
- **Remote URL:** not available (ZIP download)
- **Files inspected:** `skinematics/quat.py` (lines 1–150 and grep for scalar/convention terms)

#### Relevant logic found

**Convention confirmed SCALAR-FIRST** (`quat.py` line 733):
```python
scalar = inQuat[:,0]  # extracts index 0 as the scalar component
```
The `Quaternion` class documentation uses `[w, x, y, z]` ordering (scalar = component 0). This is the **opposite** of our SciPy `xyzw` convention where scalar is component 3 (index -1).

Class accepts inputs as: `vector` (raw array), `rotmat`, `Fick`, `Helmholtz`, `Euler` — not SciPy Rotation objects. The `unit_q()` normalization and all multiplication, inversion, and export operations assume scalar-first.

No explicit convention documentation in the module header — the convention is embedded in `q_scalar()` and is not prominently warned.

This repo has **no NaN handling** for quaternion sequences.

#### Candidate lessons for Gaga pipeline

1. **Quaternion norm sanity check**: our pipeline should assert `|q| ≈ 1.0` before and after SavGol smoothing/median filtering of quaternions. KTK's `_validate_input` pattern + scikit-kinematics convention comparison illustrates why explicit convention tests and norm checks matter.
2. **Continuity sanity check**: check for quaternion sign flips (`q` vs `-q` represent same rotation but break interpolation) before SLERP or SavGol smoothing. Scikit-kinematics doesn't address this — confirms it's a gap in external references too.

#### Applicability labels

- Quaternion norm checks: `directly_relevant`
- Quaternion continuity checks: `directly_relevant`
- Scalar-first convention itself: `not_applicable` — confirmed DIFFERENT from our xyzw convention, must not copy
- IMU/AHRS filters: `not_applicable`

#### What not to adopt

- Do NOT copy any quaternion code from scikit-kinematics.
- Do NOT use their multiplication, inversion, or SLERP implementations without explicit convention translation.
- Do NOT add Madgwick/Mahony/Kalman orientation filters to OptiTrack quaternions.

#### Concrete audit questions created

- **Q-EXT4a**: Does our pipeline assert `|q| ≈ 1.0` after loading and after each quaternion smoothing/repair step?
- **Q-EXT4b**: Does our pipeline check for and resolve quaternion sign flips (hemisphere consistency) before SLERP and SavGol quaternion filtering?

Typed dataclass with explicit defaults. All paths stored as `Path` objects (not strings). Relative paths resolved against config file location in `load_config()`. A single `load_config()` function is the only YAML entry point.

**`pipeline.py` — step gate pattern (lines 29–33):**
```python
if cfg.steps.get("c3d_export"):

```
Each stage: (1) boolean gate from config, (2) structured logging at INFO level for each file, (3) try/except with `logger.error()` per file — stage failures are logged, not raised silently.

No stage writes to a shared mutable state — each step operates on filesystem outputs.

3. **Stage-level try/except with logger**: our pipeline should not silently skip files on error — each session should be wrapped in try/except with a structured log entry.
4. **`fill_gaps_max_missing_samples = 0` as default**: our pipeline currently gap-fills by default. Adding a max-gap config parameter with a 0 = no-fill default would make this explicit.

#### Applicability labels

- Typed PipelineConfig dataclass: `candidate_for_skeleton_update`
- Per-step boolean toggle: `candidate_for_skeleton_update`
- Stage-level try/except logging: `candidate_for_logging_or_provenance`
- `fill_gaps_max_missing_samples`: `audit_question_only` (check against our current behavior)

#### What not to adopt

- Do NOT convert the current pipeline to OpenSim.
- Do NOT add IK/ID/scaling.

## 4. Cross-Repository Lessons

### L1: NaN-safe filtering is a first-class design concern

Three out of nine repositories explicitly address NaN during filtering (KTK, Pose2Sim, Vaila). Two more implicitly fail to address it (Pyomeca, Vaila filter_utils — no NaN handling). The consistent lesson is:

**NaN handling must be explicit, not assumed.** Our pipeline should log NaN counts at filter input and verify they are preserved at filter output.

The KTK pattern (detect → warn → temp fill → filter → restore NaN) is the canonical external reference. Pose2Sim's sequence-splitting approach (filter each non-NaN segment independently) is an equivalent alternative. Our pipeline uses PCHIP repair before filtering — this is a DIFFERENT operation (true repair, not temporary fill) and must be distinguished in logs.

### L2: Temporary interpolation ≠ true gap fill

KTK uses `fill_missing_samples(0)` (zero-fill) for filter stability — explicitly NOT a data repair. Our PCHIP/SLERP repair is a data repair and should be logged as such. The two must never be conflated.

Missing: in our current pipeline, there is no log field that distinguishes:
- `frames_pchip_repaired` (true gap fill — NB02)
- `frames_hampel_replaced` (artifact repair — NB04 Stage 1/2)
- `frames_nan_at_filter_input` (genuinely missing — should be preserved as NaN through Stage 3)

### L3: Config discipline — single selector + per-filter subsections + step toggles

Both Pose2Sim (Config.toml) and opensim_pyprocessing (config.py) use:
1. A single `type` selector for the active method
2. Named subsections per method with documented defaults
3. Step toggles as boolean dict

Our config does not yet have per-filter subsections. Adding them would make sensitivity studies straightforward (change `type` not the entire config structure).

### L4: Zero ≠ missing in our pipeline

Pose2Sim treats `col == 0` as missing in most filters. This is markerless-specific behavior. Confirmed we must NOT adopt this: root-relative positions, velocities at rest, and zeroed rotations are all valid zero values in our pipeline.

### L5: Quaternion convention clarity is safety-critical

Scikit-kinematics uses scalar-first `[w,x,y,z]`. NatNet/optitrack_motive uses scalar-last `[x,y,z,w]` (compatible with SciPy). The two conventions are not visually obvious from code. Any future dependency on quaternion utilities must have an explicit convention declaration.

### L6: Post-collection QC checklist

Two repositories (optitrack-main and optitrack_motive-main) provide complementary post-collection QC targets:
- Frame number continuity
- Metadata field extraction (frame rate, export rate, total frames, rotation type, length units)
- Translation plots, rotation plots, velocity sanity checks
- Marker / rigid-body availability fraction

### L7: Typed config dataclass is worth adding

Opensim_pyprocessing's `PipelineConfig` dataclass provides type safety and path resolution that our current raw YAML dict lacks. A thin typed wrapper around `config_v1.yaml` loading would catch bad config at startup, not at runtime during a long batch.
