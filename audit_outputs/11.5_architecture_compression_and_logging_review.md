## Stage Boundary Review

### S01 / S02 Boundary — KEEP SEPARATE

**Current state:** Both S01 (parse) and S02 (preprocess) use `src/preprocessing.py`. `parse_optitrack_csv()` is defined in `preprocessing.py` and called from `src/pipeline.py` at line 197.

**Conceptual justification:** The boundary IS correct despite the file co-location:
- S01 failure = file/format failure (bad CSV structure, wrong joint count, missing header)
- S02 failure = data quality failure (too many artifacts, bone length violation, gap detection issue)
- These are independent failure modes with different diagnostic implications and different recovery actions
- Independent testing requires the ability to unit-test the parser without also running gap fill logic

**File organization issue (NOT an architectural problem):** The fact that `parse_optitrack_csv()` lives in `preprocessing.py` rather than a dedicated `parse.py` is a naming inconsistency. It does not affect correctness or the stage boundary. This is `optional_improvement` for a future cleanup: either move the function to a dedicated `src/parse.py` or rename `preprocessing.py` to `src/intake.py` to reflect its dual responsibility. Neither change is required for thesis.

**Decision: KEEP SEPARATE. `optional_improvement` to rename/reorganize — defer to post-thesis.**

### S02 / S03 Boundary — KEEP SEPARATE

**Current state:** S02 produces artifact-masked arrays; S03 resamples to a common grid.

**Conceptual justification:**
- S02 operates on the native temporal grid (handles structural data quality)
- S03 changes the temporal grid itself (frame count, timestamps)
- These must be separate for the frame count provenance chain: S01 count → S03 count is the critical transition that confirms the off-by-one fix (Ticket 003) worked correctly
- If merged, the frame count comparison evidence would be lost or require explicit sub-stage logging, adding equal complexity

**Decision: KEEP SEPARATE. The off-by-one fix (Ticket 003) specifically requires this boundary.**

---

### S04 / S05 Boundary — KEEP SEPARATE; ORDERING IS CORRECT

**Current ordering:** Filter (S04) → Reference (S05).

**Why this ordering is scientifically correct:** The Markley quaternion mean is computed from T-pose frames. T-pose frames are short-duration, nearly-static windows (low velocity, near-zero acceleration). Applying Butterworth filtering to these frames before computing the Markley mean removes frame-to-frame noise in the static window, producing a more accurate quaternion mean. Computing the reference from pre-filtered data is the correct choice — it reduces the influence of marker vibration on the reference pose estimate.

**Challenge considered: Early reference eligibility QC.** Could we detect whether a usable T-pose window exists BEFORE filtering, to fail earlier? Assessment: Fast QC T3-07 (Hips variance proxy) already provides this early signal. T3-07 operates on raw data and flags sessions where the T-pose window is not detectable. No additional pre-S04 reference check is needed. The full Markley mean computation must remain post-S04.

**Decision: KEEP AS-IS. Ordering confirmed scientifically correct. Fast QC T3-07 provides early eligibility signal.**

---

### S05 / S06 Boundary — REQUIRED; MUST REMAIN SEPARATE

**Dependency chain:** S06 root-relative position computation depends on S05's reference pose (Hips-at-origin is defined relative to the T1 reference). Without S05's output, S06 cannot run. This is a hard data dependency, not a convention.

**Decision: KEEP SEPARATE. Hard dependency — S06 cannot execute without S05 output.**

---

### S06 / S11 Boundary — KEEP SEPARATE

**Conceptual justification:**
- S06 produces kinematic *state* (positions, velocities, Euler angles, CoM) — these are physical quantities directly derivable from the mocap data
- S11 produces thesis *features* (ATF, TM, D_eff, Gini) — these are derived scalars that depend on research-specific design decisions (PCA anchoring, noise floor guards, group statistics)
- This boundary protects the clean separation between "what the data shows" and "what the researcher computes from the data"
- `kinematics_master.parquet` as the interface between them is the correct design

**Decision: KEEP SEPARATE. This boundary enforces reproducibility and methodology separation.**

## Recommended Conceptual Stage Architecture

**Phase A — Data Intake and Normalization (S01, S02, S03)**

**Phase C — Feature Extraction and Validation (S11, S08, QC sidecars)**
Purpose: Produce thesis features and verify quality.

This conceptual framing is `optional_improvement` — no code changes. Useful for the thesis methods section introduction ("The pipeline operates in three phases: ..."). Does not change the implementation architecture.

## Early Reference QC Recommendation

**Decision: No additional early reference QC required beyond Fast QC T3-07.**

The question was whether S05 should be split — with a lightweight reference window eligibility check before S04, and the full Markley mean after S04.

Fast QC T3-07 (Hips variance proxy for T-pose detection) already provides early reference eligibility detection before any pipeline stage runs. If T3-07 warns, the researcher knows before running the pipeline that the T-pose window may be problematic.

Adding a second reference eligibility check between S03 and S04 would:
- Duplicate T3-07 logic
- Add stage complexity for a check that would rarely fail given T3-07 coverage
- Not improve scientific accuracy (the Markley mean must still run post-S04)

The current design — Fast QC T3-07 pre-pipeline + full reference detection post-S04 — is the minimal correct approach.

## Methodology Diagnostics Logging Contract

Phase 11's logging contract is mostly complete. The following gaps affect the ability to improve future methodology decisions.

### Gap 1 — Per-Joint Hampel Modification Rate (REQUIRED)

**What is missing:** The current Phase 11 skeleton adds `is_hampel_outlier` as a per-frame column in parquet (Ticket 011). The `filtering_summary.json` does not include a per-session Hampel activity summary.

**Why it matters for methodology decisions:**
- The Hampel filter (5-frame, 3σ) is applied with a fixed window and threshold. The decision to keep these parameters fixed was made with evidence of 0.09% modification rate across sessions.
- After all sessions are processed, the ability to review "which joints triggered Hampel activity most frequently, across all sessions" informs whether the current Hampel parameters are too aggressive, too lenient, or appropriately calibrated.
- Without per-joint rates in the stage JSON, this analysis requires post-hoc parsing of the parquet `is_hampel_outlier` column — possible but not surfaced in the session-level summary.

**Proposed addition to `filtering_summary.json`:**
```json
{
  "hampel_modification_fraction_per_joint": {
    "{joint}": 0.0
  },
  "hampel_max_fraction_any_joint": 0.0,
  "hampel_joints_above_threshold": []
}
```

**Ticket bundling:** Add to Ticket 007 (filtering_summary.json additions) or Ticket 011 (is_hampel_outlier propagation) — no new ticket required.

**Classification: `required_before_phase12` (bundle into existing ticket spec)**

### Gap 2 — Stage Runtime Logging (OPTIONAL BUT RECOMMENDED)

**What is missing:** No stage JSON currently logs execution time.

**Why it matters:**
- Ticket 015 (S04 adaptive loop) could increase filtering runtime significantly. Without a pre-Ticket-015 runtime baseline, there is no way to quantify the overhead added by the adaptive loop.
- Unexpectedly long runtimes for a specific session may indicate data quality issues (e.g., high artifact rate causing repeated Hampel iterations).
- Thesis methodology section can report "mean pipeline runtime per session."

**Proposed addition to each stage summary JSON:**
```json
{
  "stage_start_timestamp": "ISO8601",
  "stage_end_timestamp": "ISO8601",
  "stage_duration_s": 0.0
}
```

**Ticket bundling:** Each ticket that modifies a stage JSON can add these three fields. Cost: ~3 lines per stage. Total: ~18 lines across 6 stages.

**Classification: `optional_improvement` — recommend including but not blocking for Phase 12**

### Gap 3 — Gap Fill Event Count in S02 JSON (OPTIONAL)

**What is missing:** S02 JSON logs `artifact_fraction_positions` and `artifact_fraction_quaternions` (fraction of frames affected by artifact masking). It does not log gap fill events.

**Current state:** Gap fill has 0 events across all 15 sessions. The gap fill code exists (`gapfill_positions.py`) but is never triggered.

**Why it matters for the gap fill policy decision:** When sessions with genuine missing data appear (N > 15), the researcher needs to know: how many gap fill events occurred, what was their duration, which joints were affected. Without this logging, gap fill activity would be silent.

```json
  "n_gap_fill_events_quaternions": 0,
  "max_gap_duration_frames_positions": 0,
  "max_gap_duration_frames_quaternions": 0
}
```

**Ticket bundling:** Add to Ticket 009 (S02 label fix — same file, minimal additional scope) or as a standalone LOGGING_ONLY addition.

**Classification: `optional_improvement` — low cost; high future value when gap fill is eventually triggered**

### Gap 4 — Artifact Segment Length Statistics in S02 JSON (OPTIONAL)

**What is missing:** S02 JSON logs the *fraction* of frames affected by artifact masking but not the *distribution of segment lengths*.

**Why it matters for the interpolation policy decision:** The active S02 masking uses `np.interp` (linear interpolation). Linear interpolation degrades for longer artifact segments (long segments produce visible linear ramps). Knowing "maximum artifact segment length = 3 frames" vs "maximum = 25 frames" is different — the former is safe for linear interpolation; the latter may need PCHIP. Currently this evidence is not surfaced.

**Proposed addition to S02 JSON:**
```json
{
  "max_artifact_segment_frames_positions": 0,
  "mean_artifact_segment_frames_positions": 0.0,
  "n_artifact_segments_above_10_frames": 0
}
```

**Ticket bundling:** Add to Ticket 009 (S02 label fix).

**Classification: `optional_improvement`**

### Gap 5 — `fallback_path_used` Missing from session_qc_report.json Schema (REQUIRED)

**What is missing:** The Phase 11 `session_qc_report.json` schema includes `ref_is_fallback: true|false` but does NOT include `fallback_path_used`.

**Why it matters:** A reviewer seeing `ref_is_fallback: true` in the sidecar needs to know which of the 5 fallback paths was activated. Currently, this information is only in `reference_metadata.json`. The sidecar is supposed to be the single-file session QC summary — it should be self-contained for a reviewer without requiring them to open an additional stage JSON.

The `fallback_path_used` field already exists in `reference_metadata.json` (its value is read at S05 write time). Adding it to `session_qc_report.json` is a one-line addition to Ticket 014's schema specification.

**Required addition to `session_qc_report.json` schema:**
```json
{
  "ref_is_fallback": false,
  "fallback_path_used": "none|path_1_kinematic_variance|path_2_static_window|path_3_identity|path_4_previous_session|path_5_manual"
}
```

**Ticket bundling:** Update Ticket 014 schema specification in Phase 12 backlog.

**Classification: `required_before_phase12` (update Ticket 014 spec)**

### Gap 6 — S06 Stage JSON Aggregation Error Handling (REQUIRED)

**The problem:** Phase 11 introduces a new pattern: S06 reads ALL prior stage JSONs at write time to aggregate metadata into the parquet (Tickets 004 and 007). If a prior stage JSON is missing or contains malformed values, S06 could silently produce incorrect parquet metadata (e.g., `ref_is_fallback = None` in parquet instead of failing visibly).

**Required behavior:** S06 parquet write code must:
1. Attempt to read each required stage JSON (S01, S02, S04, S05)
2. If a JSON is missing: set the corresponding metadata field to `None` and log a WARNING
3. If a JSON key is missing or has an unexpected type: set the field to `None` and log a WARNING
4. Never silently propagate wrong values

**This is NOT a new ticket.** It is an implementation constraint for Tickets 004 and 007. Phase 12 must include this in the ticket specification.

**Classification: `required_before_phase12` (add to Ticket 004 and 007 specs)**

## Open Architecture Challenge

### Challenge 1: Is S06 doing too much?

S06 currently has two conceptually distinct responsibilities:
1. **Scientific computation:** quat_log angular velocity, SavGol derivatives, ISB Euler, de Leva CoM, root-relative positions, NaN guard, continuity enforcement
2. **Infrastructure aggregation:** Read all prior stage JSONs, add 5 metadata fields + 4 data columns to parquet, write quaternion diagnostic log

These could be separated into `S06a (compute)` and `S06b (write + aggregate)`. The benefit would be:
- Compute logic never touches infrastructure; infrastructure never touches compute
- Easier to test compute logic without needing all prior stage JSONs present
- Easier to re-run only the metadata aggregation without re-running kinematics

**Assessment:**
- The benefit is real but small for thesis scope (15 sessions, not 1500)
- Splitting adds a new internal interface and a new intermediate artifact
- The compute and write steps already happen in sequence within S06; the "split" is logical, not physical
- The current architecture is correct for thesis; split is worth considering for v2

**Decision: `future_v2` — Accept current S06 dual responsibility for thesis. Recommend noting the dual role in Ticket 004/007 implementation notes so implementers are aware.**

---

### Challenge 2: Is the architecture still notebook-centric?

**The concern:** A pipeline where NB01 through NB11 are each "the authoritative way to run Stage X" risks notebook-trapping — logic or parameter choices that live only in notebook cells and aren't reproducible from `run_pipeline.py`.

**Assessment for Phase 11 skeleton:**
- Phase 11 explicitly states: "Logic must live in `src/`. Notebooks call `src/` functions — they must not define new computational logic."
- The batch runner (`run_pipeline.py`) calls `src/pipeline.py` which imports from `src/preprocessing.py`, etc.
- The 5 spec fixes in v2_feature_engine (Tickets 006, 008, 010) are all in `src/v2_feature_engine.py`, not in NB11
- NB08 fix (Ticket 013) is a notebook change, but NB08 is used interactively, not in batch

**One potential issue identified:** If the QC sidecar generation (Ticket 014) is implemented in a way that it can only be triggered from a notebook, it would not be available in batch mode. Ticket 014's implementation must ensure the sidecar generation function is in `src/` (or a standalone script), callable from both `run_pipeline.py` and from a notebook.

**Decision: `required_before_phase12` — Add requirement to Ticket 014: the sidecar generation function must be callable from batch mode, not just from a notebook cell.**

### Challenge 4: Is the session_qc_report.json generation misnamed or misplaced?

In the Phase 11 architecture diagram, the QC sidecar generation appears between S06 and S11:

```
[S06 KINEMATICS] → parquet + validation_report.json
     ↓
```

## Alternative Architecture Considered

### Alternative: Merge S01+S02+S03 into "Data Preparation" module

**What it would look like:** A single `src/data_preparation.py` module that:
1. Calls `parse_optitrack_csv()`
2. Applies artifact masking
3. Applies temporal resampling
...and produces a single `data_preparation_summary.json` with substage fields.

**Arguments for:**
- Fewer files to manage
- Simpler external API (one function to call instead of three)
- A thesis-grade pipeline arguably doesn't need separate JSON files for three operations that always run sequentially without branching

**Arguments against:**
- The S03 frame count fix (Ticket 003) specifically requires that the frame count be logged at the S03 boundary. If merged, the off-by-one provenance evidence would be inside a substage log rather than at the top level of the stage summary — still available but less visible.
- Independent unit testing of parse, gap fill, and resampling logic is significantly easier with separate module boundaries.
- If future sessions require debugging (e.g., "why did this session resample incorrectly?"), the separate S03 JSON provides unambiguous evidence.
- The frame count propagation chain (S01 → S03 before and after the off-by-one fix) is a critical correctness verification that depends on the S03 boundary.

**Verdict: `reject_do_not_change`** — The separate S01/S02/S03 boundaries are justified by the frame count provenance chain, independent failure modes, and independent testability. The cost of having three stage JSONs is negligible.

---

### Alternative: Split S06 into S06a (Compute) and S06b (Write)

**What it would look like:**
- `S06a`: Run all kinematic computations; produce in-memory data structure
- `S06b`: Aggregate stage JSONs; write parquet with metadata; write validation_report.json

**Arguments for:**
- Clean separation of concerns
- Computations can be tested without needing all stage JSONs present
- Metadata aggregation can be re-run without re-running kinematics

**Arguments against:**
- Adds intermediate state management (what format does S06a output? Numpy arrays? Intermediate parquet without metadata?)
- No current failure mode requires re-running only S06b without S06a
- For N=15 sessions, the overhead of re-running S06 entirely is acceptable
- Phase 13 implementation complexity increases with two-stage S06

**Verdict: `future_v2`** — Not required for thesis. Recommended for v2 if pipeline scales to N > 50.

---

### Alternative: Early Reference Eligibility Check (pre-S04)

**What it would look like:** A lightweight `reference_eligibility_check()` function between S03 and S04 that identifies whether a T-pose window exists, without computing the Markley mean. If no window is found, stop processing before the expensive filtering stage.

**Arguments for:**
- Saves filtering time for sessions with no reference window
- Provides earlier error feedback

**Arguments against:**
- Fast QC T3-07 already does this before ANY pipeline stage
- Reference detection failure has never occurred in 15 sessions (all 5 fallback paths handled every case)
- Adding a step between S03 and S04 adds complexity for a scenario not yet observed

**Verdict: `reject_do_not_change`** — Fast QC T3-07 is the correct early signal. The current architecture is appropriate.

## Risks of Over-Consolidation

If S01, S02, and S03 were merged:

1. **Frame count provenance loss:** The off-by-one fix (Ticket 003) would no longer have a dedicated S03 frame count comparison. The provenance evidence that confirms the fix worked would be buried in a substage log.

2. **Independent test scope loss:** Tests for parse correctness would require also running gap fill and resampling. Any parse test failure would require investigation across a larger code surface.

3. **Debugging difficulty:** A session that resamples incorrectly but parses correctly would require investigating the unified module to find the failure point, rather than examining the S03 summary JSON directly.

4. **Gate architecture confusion:** gate_01_status (S01) and the S03 frame count check are different types of quality signals — duration gating vs. frame count provenance. Merging them into one stage would obscure the conceptual distinction between "session rejection" and "frame count correctness."

## Risks of Keeping Too Many Historical Stages

The current 6-stage architecture does NOT have this problem. No stage is historical-only:
- S01 has the gate (Ticket 002) — adds safety signal
- S02 has the label fix (Ticket 009) and bone QC (Ticket 007) — adds correctness

- S05 has the guards (Ticket 005) and threshold fix (Ticket 008) — adds defensive integrity
- S06 aggregates all QC metadata (Tickets 004, 007) — is the integration point

The only historical-structure smell is the `parse_optitrack_csv()` living in `preprocessing.py`, which is a file organization debt rather than a stage architecture problem.
