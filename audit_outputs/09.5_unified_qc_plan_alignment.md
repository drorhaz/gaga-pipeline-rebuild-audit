## Executive Verdict

The Unified Session QC & Validation Plan is well-designed and philosophically correct. Its three-stage architecture (Raw → Processing → Feature), its sidecar-first output policy, and its backward-compatibility principle are all aligned with the existing audit findings. The plan's own scope guardrail is honest about the risk of overengineering.

**However, roughly 60% of the plan is already covered by Phase 8 (Fast QC) or Phase 9 (Testing Plan).** The net new work for Phase 10 to evaluate is a focused set of pipeline-integrated QC enhancements at Stages 2 and 3 — specifically: propagating existing pipeline quality signals into a unified sidecar per session, and assigning feature reliability labels from that sidecar.

The plan should **not** be adopted wholesale as a Phase 10 implementation order. It should be filtered through the Phase 10 decision matrix to identify a Minimal v1 scope that adds real value without bloating the pipeline, duplicating Fast QC, or introducing instability before the core bugs from Phases 4–9 are fixed.

**Central recommendation:** The Phase 10 decision matrix must gate everything. Fix the audit findings from Phases 4–9 first (implementation Phase 13). Then add session-level QC sidecar outputs as a parallel but non-blocking workstream.

## Risks of Overengineering

The following are the **highest-risk overengineering paths** in the Unified QC Plan, ranked by impact:

### Risk 1 — Adding reliability columns per feature to kinematics_master.parquet (CRITICAL)
The plan explicitly warns against this. 803 columns already exist. Adding per-feature reliability columns would create a pattern where the parquet schema grows proportionally with the number of features, destroying its ML/DL readiness. **Sidecar is the only correct approach.**

### Risk 2 — Generating plots for every PASS session (HIGH)
Current dataset has 15 sessions. At N=50 sessions, this generates 50 × N_plots files of no automated value. Plot generation must be gated by session status. PASS sessions should produce no plots by default.

### Risk 3 — Running filter sensitivity analysis on every session (HIGH)
This would require running the Butterworth filter 3–5× per session at different cutoffs, computing PSD comparisons, and storing the results. At 50+ sessions, this is a significant compute and storage burden with marginal benefit. The REVIEW_OVERSMOOTHING finding is already UNIVERSAL (all sessions, all 57 columns). Sensitivity analysis is not needed to confirm what is already known. Classify as `offline_batch_audit_only`.

### Risk 4 — Implementing automatic cyclic anchor detection too early (HIGH)
T3-01 (T-pose detection threshold) is still DRAFT_PENDING_RESEARCH. Cyclic anchor detection is more complex than static anchor detection. Building cyclic QC before the static QC algorithm is validated would create two unvalidated systems simultaneously. No cyclic annotations exist for any current session.

### Risk 5 — Implementing day-level aggregation before session-level QC is stable (HIGH)
Day-level aggregation requires all sessions in a day to produce session_qc_reports with consistent schema and calibrated thresholds. Session-level QC has not yet been implemented or validated. Building aggregation on an unstable foundation creates compound error.

### Risk 6 — session_reliability_score formula (MEDIUM)
A numeric score (0–1) computed from a weighted combination of QC checks becomes a scientifically significant output. If the formula is wrong, biased, or not calibrated on enough sessions, it will produce false rankings. For a thesis dataset, a wrong reliability score is worse than no score. Prefer categorical labels (PASS/WARN/FAIL) until sufficient data exists to calibrate a numeric score.

### Risk 7 — Duplicating S02 bone QC logic in Stage 2.1 (MEDIUM)
S02 already computes bone QC with three systems (NB02, qc.py, forensic_report.py). Fast QC adds T3-03 on raw data. Adding another bone consistency check in Stage 2 on processed data would create a fourth system. Phase 10 should decide: is Stage 2.1 bone consistency on *processed* data genuinely different from T3-03 on *raw* data? If the answer is "not meaningfully different for this dataset," defer Stage 2.1 to future v2.

### Risk 8 — Full raw-to-processed trajectory comparison (MEDIUM)
Stage 2.2 requires aligning raw CSV positions with S06 processed parquet positions — potentially across different coordinate frames and after resampling. Phase 5.5 confirmed that 1 frame is silently lost in S03. Before this comparison can even be correct, F-INT1 must be fixed. This is a Stage 2 check that depends on a Stage 3 fix. Classify as `deep_diagnostic_qc` and defer until F-INT1 is fixed.

## User Decisions Needed Before Phase 10

### Decision 1: ROM calibration as a Hard QC gate vs. WARN-only
The Unified QC Plan proposes checking ROM calibration (`ROM_calibration_id`, `ROM_completion_status`). Historical sessions may have partial or absent ROM metadata. Several `.mcal` files are deleted in the current git status. Before Phase 10 can classify this check, the user must decide: **Is ROM calibration metadata required for session validity, or is its absence a WARN-only condition for this thesis dataset?**

Recommendation: WARN-only for existing historical sessions. Document as NOT_AVAILABLE if .mcal is absent.

### Decision 2: session_reliability_score — adopt or skip in v1?
The plan proposes a numeric reliability score (0–1). This is scientifically significant and could be misleading if not calibrated. **Should Minimal v1 use only categorical labels (PASS/WARN/FAIL) and defer the numeric score to v2, after calibration on all N sessions?**

Recommendation: Yes — use categorical labels only in Minimal v1.

### Decision 3: What triggers generation of session_qc_report.json?
Should the `session_qc_report.json` sidecar be generated **automatically for every pipeline run** (including fast PASS sessions), or **only for WARN/FAIL sessions and golden sessions**? This affects storage and the burden of maintaining the schema over time.

Recommendation: Generate for every session that completes S06. The sidecar is compact (< 5 KB per session) and avoids the problem of PASS sessions having no audit trail.

### Decision 4: Segment consistency on processed data (Stage 2.1) — add or skip in v1?
T3-03 in Fast QC already covers bone length CV on raw data. **Is there a specific need for Stage 2.1 segment consistency on *processed* parquet data?** The use case would be detecting artifacts introduced by gap filling or filtering that distort bone lengths. If no sessions have visible gap-filling distortion in current data, this adds checking infrastructure for a problem that has not been observed.

Recommendation: Skip in Minimal v1. Monitor S02 bone_qc_status instead. Promote to v2 if gap-filling artifacts become observable.

### Decision 5: Which features get reliability labels in the feature_reliability_table.csv?
The plan proposes labeling "every final downstream feature." In Minimal v1, this should be limited. **The candidate short list is: ATF (per group: axial, peripheral, transitional), TM (per endpoint), D_eff (T1-anchored and session-native), Gini (T1-anchored and session-native), AP_ratio.** Should the feature_reliability_table.csv cover all of these, or only the primary thesis metrics?

Recommendation: Cover all 5 feature types for each session in v1. The table is simple to generate from `quality_df` once the reliability labeling logic is defined.
