## Major Inconsistencies Found in Phase 12

### M1. Ticket 015 — Artifact-Fraction Control Replaces Approved PSD/Dance-Band Method

**Severity:** CRITICAL
**Phase 10 approved method:** "raise cutoff until dance-band delta ≥ -3 dB or hit ceiling" (10_rewrite_decision_gate.md §E, 10.5_phase10_correction_notes.md §Correction E)
**Phase 10.5 approved parameters:** `dance_band_threshold_db` (-3.0), `correction_step_hz` (0.5), `region_max_hz` (per-region ceilings), `max_correction_iterations` (10)

**Phase 12 instead wrote:**
> "1. Compute initial artifact fraction with static threshold
> 2. If `artifact_fraction > config.dance_band_high_water_mark`: relax threshold by step
> 3. If `artifact_fraction < config.dance_band_low_water_mark`: tighten threshold by step"

This is **artifact-fraction-based control** — a fundamentally different algorithm from the approved PSD/dance-band method. Artifact fraction measures how much data was modified; dance-band attenuation measures how much signal was preserved. They have different failure modes and different scientific semantics.

**Additional source verification finding:** `src/filtering.py` already contains an adaptive Winter residual analysis loop (lines 636–790) that selects per-joint cutoff frequencies. The adaptive loop in Ticket 015 is a **second, outer loop** that checks the PSD dance-band preservation AFTER the Winter+Butterworth filtering is complete, and adjusts the cutoff ceiling if the dance-band is over-attenuated. Phase 12 did not distinguish these two loops.

**Correction required:** Restore PSD/dance-band method. Parameters: `dance_band_threshold_db`, `correction_step_hz`, `region_max_hz`, `max_correction_iterations`. The loop checks `compute_psd_comparison()` after each `apply_adaptive_winter_filter()` call; if `psd_verdict == REVIEW_OVERSMOOTHING` (dance-band loss > threshold), it raises `region_max_hz` by `correction_step_hz` and re-runs.

**Severity:** HIGH
**Phase 10 approved:** `ref_is_fallback` → PyArrow metadata (session-level), per `10_keep_change_remove_decision_matrix.md §Approved Parquet Changes Summary`: "Metadata fields (PyArrow schema.metadata — no new data columns)"

**Phase 12 instead wrote:**

`ref_is_fallback` is constant for an entire session. Adding it as a per-row column is schema bloat — it creates ~16,000 identical boolean values per parquet file that carry zero per-frame information. It belongs in PyArrow `schema.metadata` where it can be read once per file.

**Correction required:** Remove per-row `ref_is_fallback` column. Keep PyArrow metadata field only. Total per-row data column additions from Ticket 004: 4 (session labels only).

## Filtering / Ticket 015 Correction

**Approved algorithm (Phase 10 / Phase 10.5):**

```
FOR each session:
  1. Run existing 3-stage cleaning pipeline:
     Stage 1: velocity + z-score artifact detection
     Stage 2: Hampel filter (5-frame, 3σ)
     Stage 3: Adaptive Winter low-pass (per-joint optimal cutoff via residual analysis)
     + Quaternion median filter

  2. After Stage 3, compute PSD comparison:
     dance_preservation = compute_power_in_band(filtered, dance_band=(1.0, 15.0))
     noise_attenuation = compute_power_in_band(filtered, noise_band=(20.0, 40.0))
     psd_verdict = compute_psd_comparison(raw, filtered, fs)

  3. IF psd_verdict == REVIEW_OVERSMOOTHING:
     FOR iteration = 1 to max_correction_iterations:
       Raise region_max_hz by correction_step_hz
       Re-run Stage 3 only (Hampel and artifact detection stay fixed)
       Re-check psd_verdict
       IF psd_verdict != REVIEW_OVERSMOOTHING: BREAK

  4. Log: final_cutoff_hz, n_correction_iterations, final_psd_verdict, dance_band_delta_db
```

**Approved parameters:**
- `dance_band_threshold_db`: -3.0 (default; UD-001 may adjust)
- `correction_step_hz`: 0.5
- `region_max_hz`: per-region ceilings from BODY_REGIONS dict in filtering.py
- `max_correction_iterations`: 10

**What Phase 12 got wrong:** Used `artifact_fraction` as the control variable instead of `psd_verdict`. The corrected method uses `compute_psd_comparison()` (already exists in `src/filtering.py` line 2382) and the `REVIEW_OVERSMOOTHING` verdict as the correction trigger.

**What must NOT change:**
- Stage 1 (velocity + z-score): KEEP_AS_IS
- Stage 2 (Hampel): KEEP_AS_IS
- Quaternion median filter: KEEP_AS_IS
- `src/filter_validation.py`: KEEP_AS_IS (provides the PSD analysis functions)
