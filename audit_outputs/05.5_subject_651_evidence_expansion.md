The Phase 4 S05 audit noted "0/9 fallbacks for 671" as a PASS. Subject 651 shows 2/6 sessions
with `ref_is_fallback = True`:
1. **T2_P2_R2 (DEAD):** `var_score = ‚àû` (infinity). The session has only 5 frames. The
   reference window search found no valid window and fell back to `least_motion_window_fallback`.
   The `var_score = inf` is a numeric anomaly ‚Äî division by zero or log(0) in the variance

2. **T3_P2_R2:** `var_score = 115.92` (threshold = 100.0), `mean_motion = 0.125 rad/s`.
   The subject was already moving at the start of the recording. Fallback activated. This is

**Phase 4 finding (no downstream gate on `ref_is_fallback`) is now empirically relevant:**
Both fallback sessions continue through S06 without any flag in the kinematics output.
T2_P2_R2's kinematics are computed from a fallback reference on 4 frames of data. T3_P2_R2's
kinematics are offset from an incorrect reference pose. Neither is annotated in the parquet.

**`t_pose_failed`:** `null` for both fallback sessions (not identity fallback, so t_pose_failed

## 8. New Findings from Subject 651 (not visible in subject 671)

### F-651-1: DEAD SESSION passes through all 6 pipeline stages silently

`651_T2_P2_R2` contains only 5 frames (0.03 seconds). The pipeline:
1. S01 PASS: parsed successfully, reported 5 frames, 51 segments, 0% missing
2. S02 PASS: `gate_02_status = "PASS"`, bone_qc = GOLD (cv=0.002 ‚Äî trivially low variance on
   5 frames), 0 artifacts detected
3. S03: `temporal_status = "PERFECT"`, resampled to 4 frames (‚àí1 frame loss)
4. S04: `n_frames_total = 0` (apparently filtered to 0 usable frames), `psd_verdict = "PASS"`
   with `mean_dance_delta_dB = 0.0` (vacuous ‚Äî no data to audit)
5. S05: `ref_is_fallback = True`, `var_score = ‚àû`, `time_window = [0.0, 1.0]`
6. S06: `total_frames = 4`, all 19 joints computed, `com_reliability = RELIABLE`

**No pipeline stage raised an alarm. No FAIL gate was triggered. The session appears to have
successfully processed in all batch logs.**

This is a **CRITICAL pipeline resilience gap**. A recording that is shorter than a single
dance phrase (0.03 seconds) passes all gates and produces kinematics artifacts that would be
included in any downstream analysis unless manually excluded. The minimum viable session
length must be codified as a gate, and the S01 loader should reject sessions below this
threshold (e.g., `total_frames < 360` for 3 seconds at 120Hz).

### F-651-2: `var_score = inf` in S05 reference metadata

The dead session produces `var_score = infinity` in the reference metadata JSON. This is a
numeric instability ‚Äî likely a division by zero when computing variance over a 4-frame
window. The JSON serialisation of Python's `math.inf` is `Infinity` (not valid standard JSON).
This may cause parsing errors in downstream tools that use strict JSON parsers.

### F-651-3: Bone QC CV rises progressively across timepoints (T1‚ÜíT3)

For subject 651 P2:
- T1: mean_cv = 0.351‚Äì0.356 (GOLD, 1 alert)
- T2: mean_cv = 1.011 (SILVER, 3 alerts)
- T3: mean_cv = 0.644‚Äì0.732 (GOLD, 2 alerts)

The T2 session shows notably higher bone length variability (SILVER threshold = cv > 1.0%).
This pattern was not testable in 671's data (all sessions GOLD). This suggests that
between-session marker placement reproducibility for 651 degrades at T2 and partially
recovers at T3. The CV trend should be monitored for longitudinal CoM reliability claims.

### F-651-4: Fallback reference activated in T3_P2_R2 ‚Äî first genuine fallback in dataset

`651_T3_P2_R2` is the first session in the full dataset where `ref_is_fallback = True` for a
valid (non-dead) recording. `mean_motion = 0.125 rad/s` and `var_score = 115.92` both exceed
the HIGH confidence thresholds. The subject was moving at recording start.

**Impact:** All angular kinematics for this session are computed relative to a non-neutral
reference pose (the "most static" window during movement). This introduces a systematic angular
offset for all joints. The `ref_is_fallback = True` flag is logged in S05 but NOT propagated to
the S06 `__validation_report.json` or the `__kinematics_master.parquet` metadata. A downstream
researcher would not know, from the parquet alone, that this session's angles are offset.

### F-651-5: `t_pose_failed` field is `null` (not `false`) for fallback sessions

Both `651_T2_P2_R2` and `651_T3_P2_R2` show `t_pose_failed = null` in the reference metadata.
The Phase 4 S05 audit noted that `t_pose_failed = True` is the identity fallback. The
`least_motion_window_fallback` method does not set this field at all (leaves it null), so code
checking `if t_pose_failed:` would silently pass for both non-identity fallback sessions.
